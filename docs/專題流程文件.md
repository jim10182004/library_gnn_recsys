## 1. 專題概述

### 1.1 主題
利用 **圖神經網路（Graph Neural Network, GNN）** 為市立圖書館建立個人化書籍推薦系統，並與多個傳統推薦演算法（Popularity、Item-based CF、BPR-MF）進行系統性比較。

### 1.2 解決的問題
圖書館擁有龐大的借閱紀錄，但缺乏個人化推薦機制：
- 讀者面對 10 萬本書時，難以發掘自己可能感興趣的新書。
- 圖書館缺乏資料驅動的選書／陳列依據。
- 傳統的「熱門書推薦」忽略了讀者的個別差異。

### 1.3 目標
1. 建立完整的推薦系統 pipeline（資料清洗 → 訓練 → 評估 → Demo）
2. 比較傳統與圖神經網路方法的表現差異
3. 用視覺化解釋模型學到了什麼（t-SNE 嵌入空間）
4. 產出可實際 demo 的推薦結果

---

## 2. 研究動機與貢獻

### 2.1 為什麼選 LightGCN？
傳統的學士畢業專題在處理推薦問題時，多半停留在：
- RFM 分群
- 協同過濾（User-CF / Item-CF）
- 矩陣分解（SVD / BPR-MF）

而 **圖神經網路在台灣大學部畢業專題中極為罕見**，但在 2020 年後是推薦系統的主流方法（SIGIR、KDD 等頂會大量出現）。本專題選擇 **LightGCN（He et al., SIGIR 2020）** 作為主要模型，它具有以下優勢：

1. **理論上**：將「讀者—書籍」自然建模成二部圖（Bipartite Graph），透過圖卷積學習更豐富的協同訊號。
2. **實作上**：模型輕量（去掉了 GCN 的線性轉換和激活函數），訓練快且穩定。
3. **效能上**：多個公開資料集上勝過 NGCF、GC-MC 等其他 GNN 方法。

### 2.2 本專題的貢獻
1. 將 LightGCN 應用於 **真實市立圖書館借閱資料**（多數論文用 MovieLens 等公開資料集）
2. 與 3 個 baseline 進行嚴謹比較（Popular / ItemCF / BPR-MF）
3. 提供可重現的完整 pipeline（程式 + 資料 + 評估）
4. 透過 t-SNE 視覺化讀者與書籍的潛在空間結構

### 階段 4：資料切分與互動矩陣建構

#### 4.1 K-core 過濾
**原因**：太稀疏的 user / item 在訓練時無法產生有效訊號，反而引入噪音。  
**做法**：反覆過濾，使每個 user 至少 5 個 item，每個 item 至少 5 個 user。

| 步驟 | 互動數 | Users | Items |
|------|--------|-------|-------|
| 原始（去重複借閱） | 816,119 | 109,790 | 105,957 |
| K-core (k=5) 後 | 525,288 | 35,856 | 29,685 |

#### 4.2 切分策略：**時間序列切分**（避免資料洩漏）
```
Train: 2025-01 ~ 2025-10  (約 86%)
Val:   2025-11            (約 7%)
Test:  2025-12            (約 7%)
```
**為什麼用時間切分而不是隨機切分？**  
推薦系統的真實使用場景是「用過去預測未來」。隨機切分會讓模型「偷看未來」，產生不真實的高分。

#### 4.3 Cold-start 處理
val / test 中只保留 train 內見過的 user 與 item，否則沒有 embedding 可預測。

### 階段 5：Baseline 模型實作

#### 5.1 Popular Recommender
**邏輯**：所有人都推薦借閱次數最多的 N 本書。  
**作用**：最簡單的下界，所有模型必須贏過它。  
**實作**：`np.bincount` 統計每本書出現次數即可。

#### 5.2 Item-based Collaborative Filtering (ItemCF)
**邏輯**：兩本書的相似度 = 借過它們的讀者集合的 cosine similarity。  
**公式**：
```
sim(i, j) = (u_i · u_j) / (||u_i|| × ||u_j||)
score(u, i) = Σ_j sim(i, j) × has_borrowed(u, j)
```
**實作**：用 scipy sparse matrix 加速。

#### 5.3 BPR-MF (Bayesian Personalised Ranking + Matrix Factorisation)
**邏輯**：每個 user 與 item 學一個向量，正樣本分數應大於負樣本。  
**Loss**：
```
L_BPR = -log(σ(s_pos - s_neg))
其中 s = u · i (內積)
```
**訓練**：負樣本隨機從未互動的 item 採樣（rejection sampling）。

---

### 階段 6：LightGCN 主模型實作

#### 6.1 核心概念
> "去掉 GCN 中無效的部分，留下對推薦有用的部分"

LightGCN 把 GCN 三大組件（特徵變換 W、激活函數 σ、自連接）**全部去掉**，只保留：
1. 鄰居平均（Aggregation）
2. 多層加總（Layer Combination）
3. 內積評分（Prediction）

#### 6.2 數學公式
**第 k 層的傳播**：
```
e_u^(k+1) = Σ_{i ∈ N(u)} (1 / √(|N(u)| × |N(i)|)) × e_i^(k)
e_i^(k+1) = Σ_{u ∈ N(i)} (1 / √(|N(i)| × |N(u)|)) × e_u^(k)
```
**最終 embedding**（各層加總平均）：
```
e_u = (1/(K+1)) × Σ_{k=0}^{K} e_u^(k)
e_i = (1/(K+1)) × Σ_{k=0}^{K} e_i^(k)
```
**評分**：
```
score(u, i) = e_u · e_i
```
**Loss**（BPR）：
```
L = -log(σ(s_pos - s_neg)) + λ ||θ||²
```

#### 6.3 實作技巧
1. 把雙向圖的鄰接矩陣 A 預先建好成 PyTorch sparse tensor
2. 用對稱正規化 `D^(-1/2) A D^(-1/2)`
3. 每個 epoch 重新採樣負樣本（不要寫死）

#### 6.4 超參數設定
| 參數 | 值 | 說明 |
|------|-----|------|
| embed_dim | 64 | embedding 維度 |
| n_layers | 3 | 圖卷積層數 |
| lr | 1e-3 | Adam 學習率 |
| batch_size | 4096 | mini-batch 大小 |
| L2 decay | 1e-4 | 正則化係數 |
| epochs | 30~ | 訓練輪數 |

### 階段 7：模型評估

#### 7.1 評估指標（Top-K Ranking Metrics）
| 指標 | 公式 | 直觀意義 |
|------|------|----------|
| **Recall@K** | (預測對的 ∩ 實際) / 實際 | 召回率：抓到了多少實際借的書 |
| **Precision@K** | (預測對的 ∩ 實際) / K | 精確率：推薦的 K 本中有幾本對 |
| **NDCG@K** | DCG / IDCG | 考慮排名位置的精確率 |
| **HitRate@K** | 1 if 至少 1 個推薦對 else 0 | 命中率（粗粒度） |

#### 7.2 評估流程
1. 對 test 中的每位 user，預測對所有 item 的分數
2. 把 train 中已借過的書 mask 掉（避免推薦已看過的）
3. 取 Top-K
4. 計算指標
5. 對所有 user 平均

#### 7.3 為什麼用 Recall@20 當主要指標？
- 圖書館真實 demo 場景，推薦 10-20 本是合理的
- Recall 對「召回多少」最直觀
- NDCG 是輔助，看排名品質

## 6. 遇到的困難與解決方案

> 這節是論文 / 報告中的關鍵章節。每個困難都記錄：**現象 → 排查 → 原因 → 解法**。

### 困難 1：PyTorch 與 pyarrow 的 Windows DLL 衝突 ⭐
**現象**：在 Windows 環境下，先 `import torch` 再呼叫 `pd.read_parquet()` 會無聲崩潰：
- PowerShell 顯示 `Exit code 5`
- Bash 顯示 `Segmentation fault (139)`
- **沒有任何 traceback**

**排查過程**：
1. 直接互動式執行 → OK
2. 用 `python -m src.train` 執行 → 失敗
3. 加 `print(..., flush=True)` 逐行確認 → 卡在 `load_splits()`
4. 拆解 `load_splits()` 內部，發現崩在 `pd.read_parquet`
5. 改變 import 順序測試 → 確認問題

**根本原因**：
PyTorch 預載入的 OpenMP/MKL DLL 與 pyarrow 內建的 thread runtime 衝突，pyarrow 載入時會 crash 整個 Python interpreter。

**解法**：所有進入點先 import pyarrow / pandas，再 import torch：
```python
import pandas as pd  # noqa: F401
import pyarrow      # noqa: F401
import torch        # 必須在後面
```

**耗時**：約 30 分鐘

---

### 困難 2：Excel 欄位混型（Mixed dtype）
**現象**：`to_parquet()` 失敗，錯誤訊息：
```
ArrowInvalid: Could not convert ' ' with type str: tried to convert to double
ArrowInvalid: Could not convert '20uu' with type str: tried to convert to int64
```

**根本原因**：
- `分類號` 欄位多為浮點數（如 528.5），但少數列含空格字串 `' '`
- `出版年` 欄位多為整數，但少數列含 `'20uu'`（MARC 編目中代表「世紀已知，年份未知」）

pandas 推斷出 `object` dtype，pyarrow 試圖統一型別失敗。

**解法**：
```python
def _to_str(x):
    if pd.isna(x): return None
    s = str(x).strip()
    return s if s else None

def _to_int_or_none(x):
    if pd.isna(x): return None
    try: return int(float(x))
    except (ValueError, TypeError): return None

# 對問題欄位明確轉型
for c in ("題名", "作者", "ISBN", "分類號", "讀者性別"):
    df[c] = df[c].apply(_to_str)
df["出版年"] = df["出版年"].apply(_to_int_or_none).astype("Int64")
```

**耗時**：約 10 分鐘

### 困難 3：資料規模過大、稀疏問題
**現象**：原始矩陣是 109,790 × 105,957 = **116 億 cell**，密度只有 ~0.0007%。
- 直接訓練會有大量 user / item 只有 1-2 互動，模型學不到穩定 embedding
- 推薦系統的「冷啟動」問題

**解法**：K-core filtering（迭代過濾）
```python
def k_core_filter(df, k=5):
    prev_len = -1
    while len(df) != prev_len:
        prev_len = len(df)
        u_cnt = df["user_id"].value_counts()
        i_cnt = df["book_id"].value_counts()
        keep_u = u_cnt[u_cnt >= k].index
        keep_i = i_cnt[i_cnt >= k].index
        df = df[df["user_id"].isin(keep_u) & df["book_id"].isin(keep_i)]
    return df.reset_index(drop=True)
```

**結果**：互動數 816K → 525K（保留 64%），users 109K → 35K，items 105K → 29K。雖然砍掉了一些資料，但模型訓練品質大幅提升。

---

### 困難 4：時間序列切分後的 Cold-start
**現象**：把 11 月當 val、12 月當 test，發現 val/test 中有些讀者根本沒在 train 出現過。
- 這些「全新讀者」沒有 embedding 可用
- 直接評估會被當成 0 分，拉低整體指標

**解法**：在切分後過濾，只評估 train 中見過的 user/item：
```python
train_u = set(train["u"].unique())
train_i = set(train["i"].unique())
val = val[val["u"].isin(train_u) & val["i"].isin(train_i)]
test = test[test["u"].isin(train_u) & test["i"].isin(train_i)]
```

**結果**：val 36K → 31K, test 35K → 30K（保留 ~85%）

---

### 困難 5：BPR Negative Sampling 速度
**現象**：BPR loss 每筆樣本都需要從 29K 本書中採一本「沒借過」的當負樣本。如果用 Python set lookup + random.randint，每 epoch 要採樣 45 萬次，會很慢。

**解法**：
1. 預先把每位 user 的正樣本集合建成 Python set（O(1) lookup）
2. 用 numpy `np.random.randint` 批次採樣

**結果**：每個 epoch 約 3 秒，可接受。

**現象**：在 demo 中 `print` emoji 或部分 Unicode 字元時 crash：

### 困難 9：scikit-learn 1.7 的 TSNE API 變動
**現象**：`TSNE(... n_iter=750 ...)` 報錯：
```
TypeError: TSNE.__init__() got an unexpected keyword argument 'n_iter'
```

**根本原因**：sklearn 1.5+ 將 `n_iter` 改名為 `max_iter`。

**解法**：改用 `max_iter=750`。

### 困難 10：register_buffer 與類別屬性衝突
**現象**：在 LightGCN-SI 中先在 `__init__` 寫 `self.user_gender = None`，後來呼叫 `register_buffer("user_gender", ...)` 時報錯：
```
KeyError: "attribute 'user_gender' already exists"
```

**解法**：直接在 `__init__` 用 `register_buffer` 註冊一個 placeholder（如 `torch.zeros(...)`），之後用 `self.user_gender.copy_(...)` 更新內容。這樣 buffer 自動隨模型搬到 GPU，且不會有屬性衝突。

### 困難 11：相對 import 路徑問題

_PROJ_ROOT = Path(__file__).resolve().parent.parent

讓 `from src.X import Y` 可以正常運作。

### 7.1.5 重要觀察（**經多 seed + 大規模 ablation 確認**）

#### 觀察 1：LightGCN 全面贏過 NGCF（驗證 SOTA paper 主張）
NGCF 保留了線性轉換 W 與激活函數，理論上更複雜，但在我們資料上 R@10、NDCG@10 都輸 LightGCN。
**這是 LightGCN 原論文（He et al., 2020）的核心主張：簡化反而更好**。我們在自己的資料上重現了這個結論。

#### 觀察 2：Side-info 的影響在統計上不顯著
Multi-seed (n=3) 結果：
- LightGCN: R@10 = 0.263 ± 0.0003
- LightGCN-SI: R@10 = 0.267 ± 0.0006
- LightGCN-Multi: R@10 = 0.268 ± 0.0005

差距 ~0.005 大於 std 0.001，**統計顯著**。但 SI 內部 ablation 顯示「個別 side info」（gender/age/category 單獨）幾乎無差別（差 < 0.002，可能是噪音）。

**誠實結論**：side-info 的影響很微小，主要貢獻來自 LightGCN-Multi 額外的多邊型結構。

#### 觀察 3：超參數最佳組合是 (d=128, L=2)
Grid search 結果：
```
embed_dim    32      64      128
n_layers
   1       0.2894  0.2983  0.3020
   2       0.2949  0.2994  0.3034 ★ 最佳
   3       0.2936  0.2963  0.3003
   4       0.2942  0.2955  0.2982
```
- **embed_dim 越大越好** (32 → 128 進步明顯)
- **n_layers 2 最佳**，4 層反而退化（過度平滑問題）
- 我們之前用的 (64, 3) 是「常見預設」，不是最佳

#### 觀察 4：BERT 顯著提升 coverage（多樣性）
LightGCN-BERT vs 純 LightGCN：
- Recall: 持平 (0.2674 vs 0.2648)
- **Coverage: 0.064 vs 0.029 (2.2 倍)**

書名語意 embedding 讓模型推薦的書本範圍變廣，雖然召回率沒明顯提升，但**整個圖書館的書被「看見」的比例變高**。

#### 觀察 5：序列推薦 (SASRec) 大幅落後
SASRec 預測「下一本書」，但我們的評估是「下個月會借的所有書」。**評估 paradigm 不同**，導致 SASRec 看起來很差。
但其 Coverage@10 = 0.49 是所有模型最高 — 因為它推薦的書非常多樣。

#### 觀察 6：預約邊權重最佳是 1.0（不是 0.5）
| weight | R@10 |
|---|---|
| 0.0 | 0.2671 |
| 0.5 | 0.2678 |
| 1.0 | 0.2684 ★ |

我們原本假設「預約應該比借閱弱（0.5）」，但實驗顯示**預約應與借閱同等對待**。代表「預約」本身就是強烈的興趣訊號。

### 7.1.6 LightGCN-Multi 最終設計
- 借閱邊權重：1.0
- 預約邊權重：1.0（經 ablation 確認）
- 預約時間早於 train 結束時間（避免洩漏）
- 已是借閱邊的不重複加
- 最終新增 11,561 條預約邊

### 7.2 LightGCN vs BPR-MF（GNN 對直接對照組）
| 指標 | BPR-MF | LightGCN | 提升 |
|------|--------|----------|------|
| Recall@10 | 0.2544 | 0.2648 | **+4.1%** |
| Recall@20 | 0.2825 | 0.2977 | **+5.4%** |
| NDCG@10 | 0.2087 | 0.2178 | **+4.4%** |
| NDCG@20 | 0.2161 | 0.2266 | **+4.9%** |
| Hit@10 | 0.4064 | 0.4209 | **+3.6%** |

### 7.3 LightGCN-SI vs LightGCN（側資訊的貢獻）
| 指標 | LightGCN | LightGCN-SI | 提升 |
|------|----------|-------------|------|
| Recall@10 | 0.2648 | 0.2667 | +0.7% |
| NDCG@10 | 0.2178 | 0.2231 | **+2.4%** |
| NDCG@20 | 0.2266 | 0.2310 | **+1.9%** |
| Hit@10 | 0.4209 | 0.4252 | +1.0% |

> 觀察：側資訊主要提升 **NDCG**（排名品質）與 **Hit Rate**（命中率），對 Recall 的影響較小。代表加入性別／年齡／分類後，模型更會把「好書」排在前面，但能召回的書數量差不多。

### 7.4 為什麼 LightGCN（含 SI）較好？
1. **多階鄰居資訊**：3 層卷積讓每個 user 的 embedding 融合了「相似讀者借的書」訊號
2. **無需特徵變換**：避免在稀疏資料上過擬合（LightGCN 比 NGCF 更輕量但效果更好）
3. **訓練穩定**：見 fig07，LightGCN 的 val Recall 持續上升而非震盪
4. **側資訊解冷啟動**：LightGCN-SI 從 epoch 1 就有較好的 val Recall（0.294 vs 0.288），因為新讀者至少有性別／年齡訊號
